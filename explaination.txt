Here is the complete, detailed story of how your entire project flows from the moment you write a line of code on your laptop, all the way to a user clicking "Login" on your live website. 

We can break this flow effortlessly into 4 main phases:

---

### Phase 1: You (The Developer)
1. You are on your laptop in VS Code editing files in your `frontend` or `backend` folders (like fixing CSS colors or updating database logic).
2. When you are happy with the changes, you open your terminal and run `git add .`, `git commit -m "my updates"`, and finally `git push origin main`.
3. *Your job is now completely finished.* The pipeline takes over automatically.

---

### Phase 2: The GitHub Runner (The "Middleman")
The moment GitHub receives your newly pushed code, their servers look at your `.github/workflows` file and immediately boot up a clean, temporary Ubuntu computer (a "Runner") somewhere in the cloud to do your heavy lifting.

1. **Grab the Code:** The Runner downloads your exact repository to its temporary hard drive (`actions/checkout@v3`).
2. **Setup Authentication:** The Runner needs a place to store the final apps, so it logs into **Docker Hub** using the secret credentials (`DOCKER_USERNAME` & `PASSWORD`) you securely stored in your repository settings.
3. **Build the Backend:** The Runner looks at your `/backend` folder. It downloads Node.js, runs `npm install`, packages your entire Express app inside a neat, sealed box (a Docker Image), and labels it `shuaib777sas/backend:latest`.
4. **Build the Frontend (The Magic Step):** The Runner looks at your `/frontend` file. It compiles Next.js into a production build. *Crucially*, it uses your `EC2_HOST` secret to dynamically inject your living EC2 IP address (`13.200.252.220`) directly into the Next.js Javascript files (`NEXT_PUBLIC_API_URL`). This guarantees your frontend never accidentally tries to call `http://localhost:5000` in production! It boxes this up and labels it `shuaib777sas/frontend:latest`.
5. **Publishing:** It uploads (pushes) both of these finished boxes directly up to the public Docker Hub registry so they can be downloaded from anywhere.

---

### Phase 3: The EC2 Server (The Live Deployment)
Now that your finished application is sitting safely on Docker Hub, the Runner needs to tell your live website to update itself.

1. **The Secure Connection:** The GitHub Runner grabs your raw `EC2_SSH_KEY` secret, writes it carefully into a temporary file called `key.pem`, and uses it to violently hack into—err, securely remote-login via SSH into—your live AWS EC2 server (`ubuntu@...`).
2. **Cleaning House:** Now executing commands *inside* your EC2 server, it runs `docker system prune`. This acts like a digital garbage man, instantly wiping old unused Docker images from previous deployments so your EC2 hard drive doesn't run out of space.
3. **The Download:** The EC2 server connects to Docker Hub and uses `docker pull` to download those two brand-new apps that the GitHub Runner just published a few seconds ago.
4. **The Swap:** 
   - `docker-compose down` forcefully targets your currently running live website and shuts it off (causing a split-second of downtime).
   - `docker-compose up -d` looks at the `docker-compose.yml` file sitting on your EC2 hard drive. It obediently spins up a MongoDB database, boots up your new backend on Port `5000`, boots up your frontend on Port `3000`, and links them all securely together in the background (`-d`). 

The GitHub Runner cleans up its temporary SSH key and destroys itself. The deployment is finished!

---

### Phase 4: The End User (The Result)
1. A stranger opens Google Chrome and types in `http://13.200.252.220:3000`. 
2. Your newly booted Frontend Docker container instantly serves them the Next.js visual website files.
3. The stranger types in their email and password and clicks **"Sign In"**. 
4. Because Phase 2 permanently baked your EC2 IP into the frontend code via `--build-arg`, the stranger's Chrome browser flawlessly fires a precise API request over the internet directly to `http://13.200.252.220:5000/api/auth/login`. 
5. The backend receives it, hashes the password, talks to the MongoDB container to verify the user, and sends back a highly secure JWT validation token.
6. The user is logged into the sleek blue dashboard! 

That is your entire project architecture from start to finish!
